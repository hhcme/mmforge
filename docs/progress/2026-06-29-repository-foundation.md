## Architecture Decisions

1. **Typed IDs via macro**: `NodeId`, `GeometryId`, `MaterialId`, `TextureId`, `EntityId` are newtype wrappers around `u32`, generated by a `define_id!` macro. They derive `Default`, `Serialize`, `Deserialize` for downstream use. This prevents accidentally mixing raw `u32` handles.

2. **LSM model in mmforge-core only**: The runtime model (`LsmModel`, `SceneTree`, `Geometry`, etc.) lives entirely in `mmforge-core`. No platform, GPU, or OCCT types leak into it. This satisfies the architecture contract that core must not depend on Metal/Swift/platform UI.

3. **FormatParser trait**: Defined in `mmforge-core::parser` with `detect()` and `parse()` methods. Each format parser implements this trait. Detection uses header bytes + extension, not extension alone.

4. **RenderPacket is platform-neutral**: `mmforge-render` produces `RenderPacket` containing `RenderMesh`, `RenderMaterial`, `RenderInstance`, `RenderBatch`. No Metal/D3D/Vulkan types.

5. **OrbitCamera in mmforge-render**: Camera state and math live in the render crate, not in the macOS layer. This allows all platforms to share the same camera model.

6. **macOS app uses SwiftUI document-based workflow**: `DocumentGroup` with `FileDocument` conformance. AppKit interop via `NSApplicationDelegateAdaptor`. Metal view is a placeholder (`MetalViewPlaceholder`) for Phase 1.

7. **Bridging header prepared**: `MMForge-Bridging-Header.h` is in place for Phase 1 Rust FFI declarations.

## Key Algorithms

Phase 0 does not implement business algorithms. The following algorithmic foundations are laid:

- **BoundingBox**: `extend`, `extend_point`, `center`, `half_extents`, `diagonal`, `radius`. Starts as `EMPTY` (inverted extents), grown incrementally.
- **TessellationQuality deflection**: `linear_deflection = bbox_diagonal * quality_factor`. Preview=0.002, Standard=0.0005, High=0.0001.
- **OrbitCamera**: `rotate` clamps pitch to ±89°, `zoom` uses exponential scaling, `fit` derives near/far from radius.

## Known Issues

1. **`cargo clippy` required toolchain update**: Initial `cargo clippy` failed on Rust 1.94.1 due to a `cargo-clippy` binary compatibility issue. Updated to Rust 1.96.0 which resolved it. CI uses `dtolnay/rust-toolchain@stable` so this won't affect CI.

2. **macOS app has no real functionality**: The SwiftUI shell compiles but all views are placeholders. This is expected for Phase 0.

3. **No `testfile/` directory or sample fixtures**: Phase 0 does not introduce any sample CAD files. Test fixtures will be added when parsers are implemented (Phase 1).
