# MMForge — 需求文档

> 开源工业级 2D/3D 模型解析与渲染引擎，目标替代 HOOPS Exchange + HOOPS Visualize 全套方案。
>
> 最后更新：2026-06-29

---

## 1. 项目定位

### 1.1 背景

HOOPS 是目前工业 CAD 可视化领域的事实标准（包含 Exchange 解析、Visualize 渲染、Communicator Web 三个组件），但其商业授权费用高昂，不适合中小团队和开源项目使用。

本项目旨在构建一套**完全开源、面向商业使用友好**的工业模型解析与渲染方案，覆盖从格式解析、统一运行时模型、渲染数据准备到跨平台原生渲染的完整链路。核心模块采用宽松许可证；带 GPL 约束的可选能力需要保持隔离。

### 1.2 核心目标

- **解析能力**：支持主流 2D/3D 工业格式的数据提取
- **渲染能力**：在移动端和桌面端提供高质量的模型/图纸渲染
- **跨平台**：一套核心代码，运行在 Android、iOS、Windows、macOS 上（后续扩展 Web 和鸿蒙）
- **开源**：全部代码开源在 GitHub，社区驱动

### 1.3 产品边界

- 模型编辑功能（需要完整的参数化几何内核，复杂度极高）
- 与商业 CAD 软件同等级的建模/修复能力（如 SolidWorks/CATIA 级别）
- 完整的 DWG 逆向支持（ODA 年费 $5000+，零预算下不现实）

## 2. 平台支持

### 2.1 开发顺序

| 阶段 | 平台 | 说明 |
|------|------|------|
| 第一阶段 | macOS | 先跑通 macOS 版，验证核心架构 |
| 第二阶段 | iOS | 复用 macOS 的 SwiftUI + Metal 代码（共享 90%） |
| 第三阶段 | Windows | WinUI 3 + Direct3D 12 |
| 第四阶段 | Android | Jetpack Compose + Vulkan |
| 第五阶段 | OpenHarmony | ArkUI + OpenGL ES |
| 评估中 | Web (WASM) | 先评估可行性再决定 |

### 2.2 移动端解析器策略

Rust 核心库可以编译到所有平台，但 OCCT（STEP/IGES 解析依赖）体积较大。

| 组件 | 纯 Rust | iOS | Android | 体积 |
|------|---------|-----|---------|------|
| mmforge-core | ✅ | ✅ | ✅ | < 1MB |
| mmforge-format-gltf | ✅ | ✅ | ✅ | < 1MB |
| mmforge-format-stl | ✅ | ✅ | ✅ | < 100KB |
| mmforge-format-dxf | ✅ | ✅ | ✅ | < 500KB |
| mmforge-format-step | OCCT (C++) | ⚠️ 需交叉编译 | ⚠️ 需交叉编译 | 50-100MB |
| mmforge-format-iges | OCCT (C++) | ⚠️ 需交叉编译 | ⚠️ 需交叉编译 | 同上 |

三种可选策略：

**策略 A：完整 OCCT（手机端也能解析 STEP/IGES）**
- 优点：功能完整，离线可用
- 缺点：包体积 +50-100MB，编译复杂
- 适合：对离线要求高的场景

**策略 B：纯 Rust 解析器（手机端只支持 glTF/STL/DXF/DWG）**
- 优点：包体积小，编译简单
- 缺点：不支持 STEP/IGES
- 适合：轻量级查看器

**策略 C：服务器端预转换**
- STEP/IGES 在服务器解析 → 转为 glTF → 手机端渲染
- 优点：手机端轻量，STEP 解析能力不受限
- 缺点：需要网络
- 适合：云端协作场景

> 具体策略待移动端开发时再决定。当前专注 macOS 版。

## 3. 格式支持

### 3.1 3D 格式

| 优先级 | 格式 | 解析方案 | 难度 | 说明 |
|--------|------|----------|------|------|
| P0 | STEP (AP203/AP214) | OpenCASCADE (OCCT) | ⭐⭐⭐⭐⭐ | 工业标准，必须支持 |
| P0 | glTF 2.0 | gltf-rs (Rust) | ⭐⭐ | 格式简单，生态成熟 |
| P0 | STL | 自研 | ⭐ | 格式极简，几天可完成 |
| P1 | IGES | OCCT | ⭐⭐⭐⭐ | 老格式但仍有大量存量数据 |
| P1 | OBJ | 自研或 assimp | ⭐⭐ | 简单文本格式 |
| P1 | FBX | Autodesk FBX SDK | ⭐⭐⭐ | 免费但闭源 SDK |
| P2 | 3D PDF | 自研 | ⭐⭐⭐⭐⭐ | 复杂，后期考虑 |
| P2 | JT | 逆向工程 | ⭐⭐⭐⭐⭐ | Siemens 私有，无免费方案，难度极大 |

### 3.2 2D 格式

| 优先级 | 格式 | 解析方案 | 难度 | 说明 |
|--------|------|----------|------|------|
| P0 | DXF | 自研 | ⭐⭐⭐ | 格式有公开文档，可控 |
| P1 | DWG | LibreDWG (GPL v3) | ⭐⭐⭐⭐ | 基础支持，复杂图纸可能丢实体 |
| P2 | SVG | 自研 | ⭐⭐ | XML 格式，相对简单 |
| P2 | PDF (2D) | 自研或第三方库 | ⭐⭐⭐ | 工程图纸常见格式 |

### 3.3 关于 DWG 的务实策略

DWG 是最大的坑：
- LibreDWG (GPL v3) 可用于开源项目，但只支持部分实体
- ODA 许可年费 $5000+，零预算不可用
- 自研逆向 DWG 至少 2-3 年，且有法律风险

**策略：先做 DXF（开放格式），DWG 用 LibreDWG 能支持多少算多少，不追求完美。**

## 5. 技术架构

### 5.1 核心数据模型：LSM 运行时模型

LSM（MMForge Model）是本项目的核心运行时模型，是解析层和渲染层之间的**统一契约**。早期以内存数据结构和稳定 API 为主；持久化 `.lsm` 文件格式在解析、渲染、测量、标注等契约稳定后再固化。

**定位：**
- 所有格式（STEP/IGES/glTF/DXF/DWG...）解析后统一转换为 LSM 运行时模型
- 渲染器只消费 LSM 数据，不直接依赖任何源格式
- 后续的标注、测量、编辑等功能都基于 LSM 操作

**设计原则：**
- 运行时优先：先稳定内存模型、查询接口和渲染数据契约
- 持久化后置：`.lsm` 文件格式等核心链路稳定后再冻结
- 可扩展：支持未来添加标注、测量、PMI 等附加数据
- 版本化：运行时 schema 与未来文件格式都必须支持演进

```
源文件格式解析后，统一转换为 LSM：

  STEP  ─┐
  IGES  ─┤
  glTF  ─┼──→ 解析器 ──→ LSM（统一数据模型）──→ 渲染器 ──→ 屏幕
  STL   ─┤                                    ├─ 3D: Metal / D3D12 / Vulkan
  DXF   ─┤                                    └─ 2D: Core Graphics / Direct2D / Native Canvas
  DWG   ─┘
```

| 层级 | 技术选型 | 许可证 | 说明 |

| 几何内核 | OpenCASCADE (OCCT) | LGPL 2.1 | 唯一成熟的开源 B-Rep 内核 |
| STEP/IGES 解析 | OCCT 内置 | - | 工业标准 |
| glTF 解析 | gltf-rs | MIT | Rust 原生 |
| STL/OBJ 解析 | 自研 | - | 格式简单 |
| DXF 解析 | 自研 | - | 有公开文档 |
| DWG 解析 | 可选 LibreDWG 集成 | GPL v3 | 基础支持，需与宽松许可证核心隔离 |
| macOS/iOS UI | SwiftUI | - | 苹果官方推荐 |
| macOS/iOS 3D | Metal | - | 苹果原生 GPU API |
| Windows UI | WinUI 3 | - | 微软官方推荐 |
| Windows 3D | Direct3D 12 | - | 微软原生 GPU API |
| Android UI | Jetpack Compose | - | 谷歌官方推荐 |
| Android 3D | Vulkan / OpenGL ES | - | 跨平台 GPU API |
| Rust FFI | C ABI | - | 零开销直接调用 |

### 5.3 关键技术难点

#### 5.3.1 几何内核（最大难点）

- STEP/IGES 解析不只是读文件，需要理解 B-Rep 拓扑（Face/Edge/Vertex/Loop）
- 曲面求交、曲面偏移、布尔运算是工业 CAD 核心
- OCCT 本身有 2000万+ 行代码，需要通过 FFI 绑定使用
- 移动端编译 OCCT 需要裁剪，否则体积 100MB+

#### 5.3.2 DWG 格式逆向

- 没有官方规范，ODA 花了 20 年才做到稳定支持
- LibreDWG 只支持部分实体，生产环境质量不够
- 策略：优先 DXF，DWG 做到能用就行

#### 5.3.3 原生渲染性能

- 工业模型动辄几百万三角面，移动端要流畅渲染
- 需要：实例化渲染、LOD、遮挡剔除、视锥裁剪、内存流式加载
- 各平台直接使用原生图形 API，需要在 Rust 渲染数据层统一批处理、空间索引和资源生命周期策略

#### 5.3.4 跨平台一致性

- 同一模型在不同平台上渲染结果要一致
- 字体、线宽、颜色空间都要统一
- 特别注意 Windows/macOS 的 DPI 差异

### 5.4 解析流水线

完整数据流：

```
源文件 (.step/.iges/.gltf/.stl/.dxf/.dwg)
  │
  ▼
┌─────────────────────────────────────────────┐
│  第一步：文件读取 & 格式识别                    │
│  读取文件头，判断格式版本，加载到内存             │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  第二步：格式解析（每种格式各一套）              │
│  将文件数据解析为格式特定的内存结构              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  第三步：转换为 LSM 统一数据模型                │
│  几何/拓扑/材质/结构树/元数据                   │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  第四步：Tessellation（三角化）                │
│  B-Rep 曲面 → 三角网格，生成法线/UV            │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  第五步：渲染数据准备                          │
│  VBO/IBO 打包、空间索引（BVH/AABB）           │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
              渲染器消费
```

各环节技术与工作量：

| 环节 | 技术/算法 | 依赖 | 预估工作量 |
|------|----------|------|-----------|
| 文件读取 | I/O | 无 | 1 周 |
| STEP 解析 | EXPRESS 解释器、B-Rep 映射 | OCCT | 2 周（用 OCCT） |
| IGES 解析 | 固定列格式解析、B-Rep 映射 | OCCT | 1 周（用 OCCT） |
| glTF 解析 | JSON + 二进制解析 | gltf-rs | 2 周 |
| STL 解析 | 简单文本/二进制解析 | 无 | 1 周 |
| DXF 解析 | 组码解析、实体映射 | 无 | 2-3 月 |
| DWG 解析 | 二进制逆向 | LibreDWG | 1 月（基础） |
| LSM 转换 | 数据结构映射 | 无 | 2-3 月 |
| Tessellation | 网格生成、曲面离散化 | OCCT | 2 周（用 OCCT） |
| 渲染数据准备 | VBO/IBO 打包、空间索引 | 无 | 1 月 |

详细解析器设计见 [parser/architecture.md](parser/architecture.md)。

## 8. 风险与挑战

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| OCCT 移动端体积过大 | 应用包 100MB+ | 裁剪不需要的模块，动态链接 |
| LibreDWG 质量不足 | 复杂 DWG 丢实体 | 优先 DXF，DWG 做到能用就行 |
| OpenHarmony 原生渲染成熟度 | 鸿蒙渲染路径需要验证 | ArkUI + OpenGL ES fallback，必要时延后 |
| 大模型性能 | 移动端卡顿 | LOD、遮挡剔除、流式加载 |
| 单人开发进度 | 周期可能拉长 | 先跑通完整主链路，再逐步扩展格式和平台 |

## 9. 待讨论问题

> 以下问题需要进一步讨论确认：

1. **目标客户场景** —— 主要是查看/测量/标注，还是需要编辑模型？
2. **优先支持的格式** —— 客户最常用的格式是 STEP 还是 DWG 还是其他？

5. **团队规模** —— 现在有多少人？计划招多少？
6. **许可证边界** —— 核心采用 MIT OR Apache-2.0；基于 LibreDWG 的 DWG 支持按 GPL v3 可选隔离
